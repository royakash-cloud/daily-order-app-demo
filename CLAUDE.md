# Daily Order App — Version 1 (Fresh Restart)
# New Items Finder + Availability Search

## Context — Who I Am
I am a retail pharmacist running an independent pharmacy in South

That worked for that project because it only needed monthly updates

---

## Architecture — DECIDED, do not re-open this discussion

**Option B: static site + GitHub-file persistence.**

- This is a static site hosted on GitHub Pages. No backend server,
  no database.
- **Upload and PDF parsing happen only on one fixed PC** (Windows 7,
  plain browser, no dev tools installed). PDF parsing happens
  entirely in-browser using pdf.js, reusing the 4 existing
  distributor parsers from `stock_pdf_importer.html` — do not
  rewrite these from scratch, port them.
- After parsing on the PC, the page writes a single JSON file
  (current stock + new-items diff, per distributor) to a GitHub
  repo using the GitHub REST API and a personal access token.
- **At least 2 phones need read-only access** to the same data.
  Phones load the same static page (from GitHub Pages) but never
  parse PDFs and never write anything — they only fetch the JSON
  file already written by the PC (via the GitHub API or a
  raw.githubusercontent.com URL) and render it.
- This was chosen specifically because: single writer (PC), multiple
  readers (phones), no need for a real backend, zero hosting cost,
  no credit card required anywhere in the stack.

### Token handling — decided
- A GitHub personal access token (fine-grained, scoped to only this
  one repo, contents read/write permission) is required for the PC
  to write to the repo.
- **The token is re-typed by me each session on the PC. Do not save
  it to localStorage or anywhere on disk.** Build a simple "paste
  token, use for this session only" input — once the page is closed
  or refreshed, the token should be gone and must be re-entered.
- The token must never be committed to the repo itself, and never
  appears anywhere in the phone-facing read-only code path.

## The Problem
After I upload a new stock list PDF from a wholesale distributor, I
need two things, and only these two things, in Version 1:

1. **New Items List** — on the landing page, show products that are
   newly present in the latest upload for each distributor (i.e.
   weren't in whatever was uploaded before for that distributor)
2. **Availability Search** — a search bar where I type a product
   name and see, for each of the 4 distributors, whether they have
   it and what quantity

Nothing else. No order list, no local distributors, no Firebase, no
other multi-device sync mechanism — those are explicitly OUT of
scope for this version.

---

## Distributors (v1 — exactly these 4, no others)
- Kundu
- Shivam
- Sri Ganapati
- New Ganapati

## Stock List Format
PDF upload — I do not want to manually convert PDFs to Excel first.
Each distributor's PDF has its own layout/format. I have working
JavaScript parsers already built for all 4 formats from an earlier
project (file: `stock_pdf_importer.html`) — port and reuse these,
do not rewrite from scratch:
- Kundu: numbered list format, serial + product name + qty + unit
- Shivam: clean tabular format, serial + product name + pack info
  + qty + unit
- Sri Ganapati / New Ganapati: two-column table PDF (product name,
  current stock), generated by WPS Office, requires careful
  character-gap-based space reconstruction since the underlying
  PDF text has inconsistent/missing spacing between words

## Replace Behavior
When a new stock list is uploaded for a distributor, it completely
replaces the previous one for that distributor in the GitHub JSON
file. The "new items" list is the diff between the previous upload
and this new upload for that same distributor (items present now
that weren't present before). No history beyond "previous vs
current" is needed.

## Design Requirements
- Visually simple — no decorative complexity, clear hierarchy,
  large touch targets for phone use
- Every error must be a plain-English message I can read and
  act on myself, without needing to ask for help every time
  something goes wrong
- All free tools only — no paid tiers, no credit card required
  anywhere in the stack
- The PC view (upload/parse/write) and the phone view (search/read)
  can be the same page with conditional UI, or two clearly separated
  pages — Claude Code's call, but keep it simple and explain the
  choice

## Logging Requirements — This Is Important
I want to be able to diagnose problems myself. For every operation
(PDF upload, parsing, GitHub write, search, GitHub read), log
clearly:
- What was attempted (e.g. "Parsing Kundu PDF: stock_kundu_june.pdf")
- What succeeded (e.g. "Extracted 780 products", "Wrote to GitHub
  successfully")
- What failed and why, in plain language (e.g. "Could not find any
  product lines — check that this is a Kundu-format PDF, not
  Shivam or Sri Ganapati format", "GitHub write failed — check your
  token is correct and not expired")
- Where to look: logs should be visible directly on the page
  (a simple expandable "Activity Log" panel), not just hidden in
  the browser console, since I don't routinely check dev tools

## NOT in Version 1
- Order list / cart functionality
- Local distributors (Roy, B.L, Rainbow, etc.)
- Firebase or any multi-user sync
- Stock quantity editing
- Saved/persisted GitHub tokens
- Any feature beyond: new items list + availability search
